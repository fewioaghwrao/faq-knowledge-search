## テーマ・対象ナレッジ領域

本システムが対象とするナレッジ領域は「社内業務アプリの操作手順・障害対応ナレッジ」とする。  
ログイン、権限、CSV取込、PDF出力、検索、登録・更新、メール通知、APIエラー、月次処理など、社内業務システムで発生しやすい問い合わせ・障害対応をFAQ化し、担当者が迅速に参照・活用できることを目的とする。

---

## 1. プロジェクト概要

### 1.1 背景・目的

本システムは、社内業務担当者・カスタマーサポート担当者・システム運用担当者が、FAQ・手順書・障害対応メモを検索し、業務上の疑問やエラー対応方法を素早く確認するためのナレッジ検索システムである。

フェーズ1では基本的なFAQ管理・キーワード検索を実装する。フェーズ2以降では検索精度の向上・AI要約（RAG）・権限管理・運用ログの強化を段階的に追加し、実務利用に耐える社内ナレッジ基盤として発展させる。

### 1.2 フェーズ構成

| フェーズ | 主な内容 | 備考 |
|---|---|---|
| フェーズ1 | FAQ管理・基本検索・タグ/カテゴリ管理・JWT認証 | 本要件定義のベース |
| フェーズ2 | 高度なキーワード検索（スコア順・複数フィールド・複数キーワードAND） | AIなし |
| フェーズ3 | AI要約回答（RAG：検索結果をAIに渡して要約） | 初のAI統合 |
| フェーズ4 | AI回答への参照元FAQ表示・幻覚対策 | 根拠の可視化 |
| フェーズ5 | AI実行履歴の保存・管理画面への表示 | 運用ログ強化 |
| フェーズ6 | ロールベースアクセス制御（RBAC）強化 | 権限の細分化 |
| 余力追加 | 公開/非公開・閲覧数・評価・CSVインポート・Markdown・通知等 | 優先度Low |

### 1.3 想定利用者

#### 一般利用者

| ユーザー | 主な使い方 |
|---|---|
| 経理担当者 | 請求処理・CSV取込・月次締めの手順を確認する |
| カスタマーサポート | 顧客からの問い合わせに対してFAQを検索・AI回答を参照する |
| 社内ヘルプデスク | ログイン障害・権限エラー・PDF出力不具合の対応手順を探す |
| 業務システム運用担当 | API障害・取込エラー・エラーパターンの過去対応を確認する |
| 新人・引き継ぎ担当者 | 業務手順や障害対応メモを学ぶ |

#### ナレッジ管理者

FAQ・カテゴリ・タグを登録・編集・削除するユーザー。AI実行履歴や検索履歴・アクセス統計を確認し、FAQ改善を行う。

#### システム管理者

ユーザー管理・権限管理・公開状態の管理・システム設定を担当するユーザー。

### 1.6 用語定義

| 用語 | 定義 |
|---|---|
| FAQ | よくある質問と回答のセット。タイトル・本文・カテゴリ・タグで構成される |
| カテゴリ | FAQを業務領域で分類するグループ（例：請求処理、CSV取込） |
| タグ | FAQに付与するキーワードラベル。複数付与可能 |
| 検索履歴 | ユーザーが入力したキーワードの履歴。FAQ改善のために参照する |
| 公開/非公開 | FAQの表示状態。非公開のFAQは一般利用者に表示されない |
| ナレッジ管理者 | FAQの登録・編集・管理を担当するロール |
| AI要約回答 | RAGによりDBから取得した関連FAQをAIが要約して返す回答 |
| 参照元FAQ | AI要約の根拠として提示される元FAQのリスト |
| AI実行履歴 | AI要約回答の実行ログ（質問・回答・参照元・日時・ユーザー・成否） |
| スコア順 | キーワードの一致度・出現頻度に基づいた検索結果の優先順位 |
| ApplicationUser | ASP.NET Core Identityの`IdentityUser`を継承した拡張ユーザークラス。`AspNetUsers`テーブルに対応し、DisplayName・IsActive等の業務固有カラムを追加する |
| AspNetRoles | Identityが自動生成するロール定義テーブル。User・Editor・Adminの3ロールを管理する |
| AspNetUserRoles | ユーザーとロールの中間テーブル。Identityが自動生成する |
| UserId | ApplicationUserのId（GUID）。AiHistories・SearchHistories・FaqRatingsで実行者・操作者を識別するための外部キー |

### 2.2 機能詳細

#### F-101〜F-103: 高度なキーワード検索（フェーズ2）

| 項目 | 内容 |
|---|---|
| 概要 | タイトル・本文・カテゴリ名・タグ名を横断してキーワード検索し、スコア順に結果を表示する |
| アクター | 一般利用者・ナレッジ管理者 |
| 事前条件 | FAQが1件以上登録されていること |
| 基本フロー | ①検索バーにキーワードを入力（例：「CSV 取込 エラー」）→ ②スペース区切りでAND検索 → ③タイトル・本文・カテゴリ・タグを横断検索 → ④スコア順（一致度・出現頻度）で結果を表示 |
| 表示例 | 1. CSV取込でエラー行が発生した場合の確認手順（スコア高）2. CSVの文字コードエラーへの対応 3. 必須列不足エラーの修正方法 |
| ビジネスルール | 検索対象フィールドの優先度：タイトル > タグ名 > カテゴリ名 > 本文。公開中FAQのみ対象（管理者は全件） |

#### F-201〜F-204: AI要約回答（フェーズ3）

| 項目 | 内容 |
|---|---|
| 概要 | ユーザーが自然文で質問を入力すると、関連FAQを検索してAIが要約回答を生成する |
| アクター | 一般利用者・ナレッジ管理者 |
| 基本フロー | ①ユーザーが質問文を入力 → ②キーワード抽出・FAQ検索で上位3〜5件を取得 → ③取得FAQをコンテキストとしてAIに渡す → ④AIが要約回答を生成 → ⑤回答と参照元FAQを表示 |
| AIの役割（許可） | 検索で見つかったFAQをもとに利用者向けにわかりやすく要約する |
| AIの役割（禁止） | 根拠がないことを断言する・FAQにない手順を勝手に作る・機密情報を生成する・問い合わせ先や責任者を勝手に決める |
| プロンプト制御例 | システムプロンプトに「提供されたFAQの内容のみを根拠として回答してください。FAQに記載のない情報は回答に含めないでください」を含める |
| AI呼び出し先 | Azure OpenAI Service（GPT-4o）または Anthropic Claude API（設定で切替可能） |

#### F-301〜F-303: 参照元FAQ表示（フェーズ4）

| 項目 | 内容 |
|---|---|
| 概要 | AI要約回答の下部に、その回答の根拠となったFAQのタイトルとリンクを必ず表示する |
| 表示形式 | AI回答テキストの下に「参照元：」セクションを設け、FAQ名とリンクを箇条書きで表示する |
| 幻覚対策 | AI回答の冒頭または末尾に「この回答はFAQをもとに生成されています。必ず参照元を確認してください」の注意文を固定表示する |
| ビジネスルール | 参照元が0件の場合は「該当するFAQが見つかりませんでした。管理者にご相談ください」を表示し、AIによる回答生成は行わない |

#### F-401〜F-404: AI実行履歴（フェーズ5）

| 項目 | 内容 |
|---|---|
| 保存項目 | 質問文・検索に使ったキーワード・AI回答・参照元FAQ（ID・タイトル）・実行日時・実行ユーザー（UserId。未ログイン時はSessionId）・成功/失敗・エラーメッセージ |
| 一覧表示 | 管理者が実行日時・ユーザー・質問文で絞り込み検索可能。20件/ページでページネーション表示 |
| 詳細表示 | 個別ログの質問文・AI回答全文・参照元FAQ一覧・評価結果を表示する |
| フィードバック記録 | ユーザーが「役に立った（👍）/立たなかった（👎）」を選択するとAI実行履歴レコードに評価値を記録する |

#### F-501〜F-504: ロールベースアクセス制御（フェーズ6）

| ロール | 定義 | 追加権限 |
|---|---|---|
| User（一般利用者） | FAQ閲覧・キーワード検索・AI質問・自分の検索履歴閲覧 | — |
| Editor（ナレッジ管理者） | Userの全権限 + FAQ登録・編集・削除・カテゴリ/タグ管理・AI実行履歴閲覧 | フェーズ6で追加 |
| Admin（システム管理者） | Editorの全権限 + ユーザー管理・ロール変更・システム設定 | フェーズ6で追加 |

### 5.2 テーブル詳細

#### AspNetUsers（ユーザーテーブル / ApplicationUser拡張）【フェーズ1】

ASP.NET Core Identityが自動生成する `AspNetUsers` テーブルを `ApplicationUser : IdentityUser` として継承・拡張する。Identity標準カラム（Id・Email・PasswordHash・LockoutEnabled等）に加え、以下の業務固有カラムを追加する。

| No | カラム名 | データ型 | NULL | キー | デフォルト | 備考 |
|---|---|---|---|---|---|---|
| 1 | Id | nvarchar(450) | NOT NULL | PK | | IdentityのデフォルトGUID文字列 |
| 2 | DisplayName | nvarchar(100) | NOT NULL | | | 画面表示名 |
| 3 | IsActive | bit | NOT NULL | | 1 | 有効=1 / 無効=0。無効ユーザーはログイン不可 |
| 4 | CreatedAt | datetime2 | NOT NULL | | GETUTC | アカウント作成日時（UTC） |
| 5 | UpdatedAt | datetime2 | NULL | | | アカウント更新日時（UTC） |
| — | UserName | nvarchar(256) | NULL | | | Identity標準（メールアドレスと同値で運用） |
| — | Email | nvarchar(256) | NULL | UQ | | Identity標準 |
| — | PasswordHash | nvarchar(max) | NULL | | | Identity標準（PBKDF2ハッシュ） |
| — | LockoutEnabled | bit | NOT NULL | | 1 | Identity標準（ログイン失敗ロック） |
| — | AccessFailedCount | int | NOT NULL | | 0 | Identity標準（失敗回数） |

## 6. 非機能要件

| 項目 | 要件 |
|---|---|
| パフォーマンス | FAQ一覧・検索のレスポンスタイム2秒以内（同時接続10ユーザー想定）。AI要約は30秒以内（タイムアウト設定） |

| 可用性 | 開発・検証用のため高可用性要件は設けない。バックアップはAzureの標準機能を利用 |

| アクセシビリティ | 社内ツールのため必須要件としないが、基本的なキーボード操作を可能とする |
| AI利用コスト管理 | AI APIコール数・トークン数を月次でモニタリング。過剰利用を防ぐためレート制限を設ける（フェーズ3以降） |

### 10.2 AI要約のガードレール設計（プロンプト例）

```
【システムプロンプト】
あなたは社内FAQ検索システムのアシスタントです。
以下のルールを厳守してください。

【許可する回答】
- 提供されたFAQの内容をもとに、利用者向けにわかりやすく要約した回答
- FAQに記載された手順・確認方法の説明

【禁止する回答】
- 提供されたFAQに記載のない情報の断言
- FAQにない手順・操作方法の独自生成
- 問い合わせ先・責任者・担当部署の推測や断言
- 機密情報・個人情報を含む内容の生成
- 「必ずしてください」「絶対に〜です」等の断定的な保証表現

【回答の末尾】
必ず「詳細は参照元FAQをご確認ください。」を付記してください。

【FAQコンテキスト】
{context}

【ユーザーの質問】
{question}
```

### 10.3 検索スコアリング設計（フェーズ2）

| フィールド | スコア重み | 理由 |
|---|---|---|
| タイトル（完全一致） | 1.0 | 最も関連度が高い |
| タイトル（部分一致） | 0.8 | 高い関連度 |
| タグ名（一致） | 0.7 | ナレッジ管理者が付与したキーワード |
| カテゴリ名（一致） | 0.5 | 業務領域レベルの一致 |
| 本文（部分一致） | 0.3 | テキスト量が多いため重みを低くする |

複数キーワードの場合は各キーワードのスコアの合計値を最終スコアとし、降順で表示する。
